« Garder la mémoire, éclairer les parcours. » — Application web de suivi du devenir des cohortes d'étudiants de l'IUT de Villetaneuse, construite en S3 puis reprise et améliorée en S4.

Contexte du projet

L'IUT forme selon le diplôme BUT (Bachelor Universitaire de Technologie), organisé par compétences. Le suivi du devenir des cohortes — qui passe, qui redouble, qui se réoriente, qui est diplômé — est un vrai besoin institutionnel. Mnémosyne répond à cette question en visualisant le parcours d'une promotion, de l'entrée en BUT1 jusqu'à la sortie, à partir des données de l'API ScoDoc et dans le respect du RGPD (données anonymisées). Dans la mythologie grecque, Mnémosyne est la déesse de la mémoire : garder la trace du passé pour éclairer l'avenir.

Du S3 au S4 : deux temps

🏗️

S3 — Construction

Première version de l'application : architecture MVC en Flask (controllers / model / service), base SQLite, récupération des données via l'API ScoDoc et un premier diagramme de Sankey (un seul niveau). Fonctionnel, mais perfectible : décisions et états stockés en TEXT dans une seule table, Sankey peu lisible, pas de charte ni de README.

S4 — Amélioration

SAÉ S4 « Développement d'une application » : reprendre un projet S3 existant pour l'améliorer. Notre équipe a analysé puis choisi une base de départ, avant de refondre la base de données et d'enrichir le diagramme de Sankey pour rendre les parcours bien plus lisibles.

Phase 1 (S4) — Analyse de l'existant

Avant de coder, nous avons comparé quatre projets Mnémosyne issus du S3 selon quatre critères, afin de choisir la meilleure base de départ. Le critère du langage était décisif : la ressource R3.01 nous avait formés à Flask (Python), et le semestre étant court, reprendre un projet en PHP aurait imposé d'apprendre le langage en plus du code.

Décision finale — un choix argumenté

Nous n'avons pas pris le projet le mieux noté (Solene0971, 17/20) mais le nôtre (16/20) : il nous devançait surtout sur la facilité de reprise, puisque nous en sommes les auteurs et en maîtrisons déjà l'architecture. Nous avons toutefois gardé « le meilleur de chacun » : la modélisation de la base de Solene0971, l'idée d'une table SCENARIO de Mohamed-devhub (pour gérer les cas de passage, ex. BUT2 FI → BUT3 FA) et la légèreté de la page d'accueil.

Phase 2 (S4) — Améliorations apportées

1

Refonte de la base de données

Nouveau modèle entité-association et script de création : ajout d'un référentiel et des compétences, là où l'ancienne base stockait décisions et états dans un simple champ TEXT.

2

Table SCENARIO

Gestion explicite des cas de passage entre formations et régimes (FI / FA), pour comptabiliser correctement chaque étudiant.

3

Diagramme de Sankey multi-niveaux

Flux de cohorte complet : BUT1 → décisions → BUT2 → … → BUT3 → Diplômé / Redoublement / Abandon. Blocs déplaçables, palette de la charte (trois bleus BUT, décisions en doré), légende à icônes et cartes d'indicateurs.

4

Interface & charte graphique

Barre de navigation soignée, logos Sorbonne Paris Nord + IUT en pied de page, pages d'accueil et de connexion tenant sur une seule fenêtre, accueil allégé.

Difficultés & leçons tirées

Analyse réflexive — l'angle critique

Horizon des données : les étudiants de BUT1 INFO n'apparaissaient qu'à partir de 2023 — j'ai appris à vérifier la période réellement couverte avant toute analyse.
Compétences ↔ notes : aucun identifiant ne les reliait directement ; il a fallu une correspondance basée sur leur position.
Cohérence : le modèle décrit dans le rapport ne correspondait pas toujours à la base réelle, d'où plusieurs ajustements.
Reprise de code : repartir d'un projet existant fait gagner du temps, mais oblige aussi à corriger les choix faits auparavant.
